# 通用开发规范

> 适用于前后端所有代码，AI 生成代码必须遵守

## 必须遵守（放开头，AI 最敏感）

### 命名规范
- 变量/函数：`camelCase`
- 常量：`UPPER_SNAKE_CASE`（如 `MAX_FILE_SIZE = 50 * 1024 * 1024`）
- 类/接口/类型：`PascalCase`
- 文件名：`kebab-case`（路由/工具）或 `camelCase.ts`（服务/模型）
- 禁止使用拼音、无意义缩写（如 `getUsrInfo` → `getUserInfo`）

### 代码风格
- 2 空格缩进，禁止混用 Tab 和空格
- 行宽 ≤ 120 字符，超长则换行
- 使用单引号，不使用双引号
- 语句末尾加分号
- 对象/数组尾随逗号（trailing comma）

### 注释规范
- 所有导出函数/方法必须有 JSDoc 注释（`/** ... */`）
- 复杂业务逻辑必须有中文行内注释
- 禁止注释掉的大段代码，用 Git 历史管理
- 临时方案用 `// TODO: 说明`，已知问题用 `// FIXME: 说明`

## 提交规范
- Commit Message 格式：`[类型] 描述`
- 类型：`feat`（新功能）/ `fix`（修复）/ `docs`（文档）/ `style`（格式）/ `refactor`（重构）/ `test`（测试）/ `chore`（工具）
- 示例：`[feat] 新增地图标点功能`、`[fix] 修复 DXF 解析坐标偏移问题`

## 禁止行为（强制）
- ❌ 禁止硬编码魔法值（提取为常量或配置）
- ❌ 禁止打印敏感信息（密码、token、密钥）
- ❌ 禁止空 catch 块（至少记录日志或抛出）
- ❌ 禁止直接修改 Spec 文件外的代码（SDD 流程：先改 Spec → 再让 AI 同步代码）
- ❌ 禁止使用 `any` 类型（除非确实无法推断，需加注释说明原因）
- ❌ 禁止 `console.log` 进入生产代码（使用 logger）
- ❌ 禁止在循环中执行数据库/网络请求（批量操作）

## 错误处理
- 所有异步操作必须用 try-catch 包裹
- 错误必须记录日志（含堆栈信息）
- 向用户返回友好错误消息，不暴露内部实现细节
- 使用自定义错误类区分业务错误和系统错误

## AI 生成代码要求
- 生成的代码必须可编译、可运行
- 必须遵循项目现有代码风格和目录结构
- 必须包含必要的错误处理和边界条件检查
- 必须生成对应的单元测试
- 修改代码前先确认是否符合 Spec，不符合则先更新 Spec

## 安全规范
- 所有用户输入必须校验和过滤
- 文件上传必须校验类型和大小
- API 接口必须防止 SQL 注入、XSS、CSRF
- 敏感配置使用环境变量，禁止硬编码
- 日志中禁止记录用户密码、token 等敏感信息

---

## 重要提醒（放结尾，AI 最敏感）

- **Spec is Truth**：代码是 Spec 的实现，修改先改 Spec
- **单一职责**：一个函数/类只做一件事
- **DRY 原则**：重复代码超过 2 次必须提取
- **防御性编程**：假设外部输入不可信，做好校验
