**状态**: 待实施

## 1. 问题描述

### 1.1 现状
- 白坪.dxf 解析出 **21,666 个实体**
- 全量渲染导致：
  - ❌ 接口响应慢（>10 秒）
  - ❌ 前端渲染卡顿
  - ❌ 内存占用高
  - ❌ 地图操作不流畅

### 1.2 用户需求
- ✅ 根据地图缩放级别自动调整显示精度
- ✅ 只渲染可见区域，保证流畅性
- ✅ 保持图形/点位连贯性
- ✅ 放大看到细节，缩小看到轮廓

## 2. 解决方案

### 2.1 技术架构

```
┌─────────────────────────────────────────────────────────┐
│                     前端 (Leaflet)                       │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐           │
│  │视口检测   │  │LOD 控制   │  │数据简化   │           │
│  │(可见区域) │  │(zoom 级别)│  │(Douglas)  │           │
│  └───────────┘  └───────────┘  └───────────┘           │
└─────────────────────────────────────────────────────────┘
                          ↕ API 请求
┌─────────────────────────────────────────────────────────┐
│                   后端 (Node.js)                         │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐           │
│  │LOD 过滤   │  │视口裁剪   │  │简化算法   │           │
│  │(zoom 级别)│  │(bounds)   │  │(Douglas)  │           │
│  └───────────┘  └───────────┘  └───────────┘           │
│                                                          │
│  ┌──────────────────────────────────────────┐           │
│  │         原始 CAD 数据 (21,666 实体)         │           │
│  └──────────────────────────────────────────┘           │
└─────────────────────────────────────────────────────────┘
```

### 2.2 核心优化策略

#### 策略 1: LOD (Level of Detail) 分层

| Zoom Level | 返回比例 | 实体数 | 用途 |
|------------|----------|--------|------|
| zoom < 10 | 1% | ~217 | 省级视图，只看轮廓 |
| zoom 10-12 | 5% | ~1,083 | 市级视图，主要结构 |
| zoom 12-14 | 20% | ~4,333 | 区级视图，中等细节 |
| zoom 14-16 | 50% | ~10,833 | 街道视图，较多细节 |
| zoom > 16 | 100% | 21,666 | 建筑视图，全部细节 |

#### 策略 2: 视口裁剪 (Viewport Culling)

```typescript
// 只渲染可见区域的实体
const visibleEntities = entities.filter(entity => {
  return map.getBounds().contains(toLatLng(entity));
});

// 添加缓冲区（预加载）
const bufferBounds = map.getBounds().pad(0.2); // 外扩 20%
```

#### 策略 3: 数据简化 (Douglas-Peucker 算法)

```typescript
// 简化多段线，保持形状但减少点数
const simplified = simplifyPolyline(vertices, tolerance);

// tolerance 根据 zoom level 调整
const tolerance = calculateTolerance(zoom);
// zoom 越小，tolerance 越大，简化越多
```

### 3.2 接口设计

#### 修改现有接口

```typescript
POST /api/cad/merge-chunks

请求:
{
  "chunkId": "xxx",
  "filename": "白坪.dxf",
  "totalChunks": 4,
  "renderOptions": {      // 新增：渲染选项
    "zoom": 14,           // 当前地图缩放级别
    "bounds": {           // 当前视口范围
      "north": 39.95,
      "south": 39.85,
      "east": 116.45,
      "west": 116.35
    },
    "simplify": true,     // 是否简化
    "maxEntities": 5000   // 最大返回实体数
  }
}

响应:
{
  "code": 200,
  "data": {
    "filename": "白坪.dxf",
    "layers": [...],
    "metadata": {
      "totalEntities": 21666,      // 总实体数
      "returnedEntities": 2166,    // 实际返回
      "simplified": true,          // 是否简化
      "zoomLevel": 14,             // LOD 级别
      "simplifyTolerance": 0.001   // 简化容差
    }
  }
}
```

#### 新增接口

```typescript
// 获取 LOD 配置
GET /api/cad/lod-config

响应:
{
  "lodLevels": [
    {"zoom": 10, "ratio": 0.01, "tolerance": 0.01},
    {"zoom": 12, "ratio": 0.05, "tolerance": 0.005},
    {"zoom": 14, "ratio": 0.20, "tolerance": 0.001},
    {"zoom": 16, "ratio": 0.50, "tolerance": 0.0005},
    {"zoom": 18, "ratio": 1.00, "tolerance": 0}
  ]
}

// 动态加载实体（地图移动/缩放时）
POST /api/cad/entities/query

请求:
{
  "filename": "白坪.dxf",
  "zoom": 14,
  "bounds": {...}
}

响应:
{
  "entities": [...],  // 过滤后的实体
  "count": 2166
}
```

## 5. 测试用例

### 5.1 功能测试

#### 测试 1: LOD 分级
```
步骤:
1. zoom=10，上传白坪.dxf
2. zoom=14，重新加载
3. zoom=18，重新加载

预期:
- zoom=10: 返回~217 个实体 (1%)
- zoom=14: 返回~4,333 个实体 (20%)
- zoom=18: 返回 21,666 个实体 (100%)
```

#### 测试 2: 视口裁剪
```
步骤:
1. 地图移动到北京
2. 上传白坪.dxf
3. 只渲染北京范围内的实体

预期:
- 只返回视口内的实体
- 视口外的实体不渲染
```

#### 测试 3: 数据简化
```
步骤:
1. zoom=10，简化多段线
2. zoom=18，不简化

预期:
- zoom=10: 多段线点数减少 90%
- zoom=18: 保持原始点数
- 形状保持连贯
```

### 5.2 性能测试

